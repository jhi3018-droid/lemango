## Area 1: Firestore 보안 규칙

**종합 등급**: 🔴
**한 줄 요약**: 모든 컬렉션이 "로그인된 사람 누구나 무엇이든 가능"에 가깝습니다. 등급(grade) 기반 권한이나 작성자 본인 확인은 클라이언트 화면에서만 막혀 있으며, 서버(Firestore 규칙) 차원에서는 거의 통제가 없습니다.

### 발견사항

- 🔴 **[심각] `firestore.rules:8-10` — `sharedData` 컬렉션 무방비**
  "상품 데이터", "행사일정", "업무일정", "신규기획", "설정", "판매채널", "색상 마스터" 등 회사 운영의 거의 모든 데이터가 이 하나의 컬렉션에 들어 있습니다. 현재 규칙은 "로그인된 사람 누구나 무엇이든 가능"입니다. 직급이 가장 낮은 담당자(grade 1)도 콘솔만 열면 `상품_전체_데이터`나 `색상마스터`를 한 줄 명령으로 통째로 지울 수 있습니다.

- 🔴 **[심각] `firestore.rules:13-15` — `notifications` 권한 부재**
  남의 알림을 임의로 수정·삭제할 수 있습니다. 코드는 본인 알림만 보여주지만, 규칙은 그 가정을 지키지 못합니다.

- 🔴 **[심각] `firestore.rules:18-20` — `editLocks` 잠금 우회 가능**
  편집 잠금은 다른 사람이 같은 상품을 동시에 수정하지 못하게 하는 장치인데, 누구나 남의 잠금을 풀 수 있어 사실상 잠금이 의미가 없어집니다.

- 🔴 **[심각] `firestore.rules:37-42` — `leaves`(연차) / `firestore.rules:52-57` — `attendance`(출퇴근)**
  연차 신청 기록과 출퇴근 기록을 다른 직원이 수정할 수 있습니다. 인사 데이터의 무결성을 보장할 수 없습니다.

- 🔴 **[심각] `firestore.rules:86-91` — `personalSchedules` 본인 확인 없음**
  본인의 개인일정뿐 아니라 다른 사람의 개인일정도 자유롭게 만들고 지울 수 있습니다.

- 🟡 **[주의] `firestore.rules:78-83` — `comments` 삭제 규칙과 클라이언트 코드 불일치**
  `comments.js:316`은 작성자 OR grade>=3 관리자 양쪽 모두에게 삭제 권한을 주지만, Firestore 규칙(82행)은 작성자(`resource.data.uid == request.auth.uid`)만 허용합니다. 관리자가 댓글을 지우려 하면 "권한 없음" 오류로 실패합니다.

- 🟡 **[주의] `firestore.rules:5` — `isAuth()` 함수가 승인(`approved`) 여부를 보지 않음**
  Firebase Auth 계정만 있으면 통과합니다. 가입 대기 상태(`pending`)인 사용자가 로그인 화면을 우회하고 콘솔에서 직접 데이터에 접근하는 것을 막을 수 없습니다. (Area 2 참조)

- 🟡 **[주의] `storage.rules:7,13` — Storage 첨부파일 본인 확인 없음**
  게시판 첨부파일과 기획 임시 이미지의 업로드/삭제가 "로그인된 누구나" 가능합니다. 자기가 올린 파일이 아니어도 삭제할 수 있어 다른 사람의 첨부파일을 임의로 지울 수 있습니다.

- 🟢 **[안전] `users` 컬렉션의 update/delete 규칙**
  본인 정보는 본인이, 다른 사람 정보는 grade>=3만 수정 가능하도록 잘 제한되어 있습니다. 삭제는 누구도 못 하도록 `delete: false`로 막혀 있습니다.

- 🟢 **[안전] `salaries`(급여) 컬렉션**
  본인 또는 관리자(grade>=3)만 읽고, 쓰기·삭제는 관리자만 가능하도록 엄격하게 설정되어 있습니다.

- 🟢 **[안전] `activityLogs`** — 한 번 생성된 로그는 수정·삭제 불가로 잘 잠겨 있습니다.

- 🟢 **[안전] `backups`** — 시스템 관리자(grade>=4)만 쓰기·삭제 가능하도록 잘 보호되어 있습니다.

## Area 2: 인증 상태

**종합 등급**: 🟡
**한 줄 요약**: 로그인 화면을 우회하지 못하도록 클라이언트 흐름은 잘 짜여 있으나, "가입 대기 사용자가 콘솔로 직접 데이터에 접근하는 경로"는 막혀 있지 않습니다.

### 인증 흐름 요약

- 인증 방식: Firebase Auth 이메일/비밀번호 (`js/auth.js:85`)
- 세션 지속: `SESSION` (`js/auth.js:20`) — 탭을 닫으면 자동 로그아웃. 보안상 적절한 선택.
- 로그인 게이트: `js/main.js:11-12`에서 `State.currentUser`가 없으면 `initApp()`을 호출하지 않아 데이터 로드 자체가 안 됨. 클라이언트 흐름상 견고함.
- 승인 흐름: `js/auth.js:140-149` — Firestore의 `users.status !== 'approved'`이면 즉시 `signOut()` 후 안내.
- 등급 권한: `js/core.js:685-699` `TAB_PERMISSIONS` + `canAccessTab()` 함수로 클라이언트 측 탭 접근을 차단. `js/main.js:320` `updateTabVisibility()`로 메뉴 버튼도 숨김.

### 발견사항

- 🔴 **[심각] `firestore.rules:5` + `js/auth.js:140` — 승인 확인이 클라이언트에만 존재**
  `checkApproval()`이 `status !== 'approved'`이면 `signOut()`을 호출하지만, 그 순간 이미 사용자는 Auth 토큰을 보유 중입니다. 악의적 사용자가 브라우저 콘솔에서 `firebase.firestore().collection('sharedData').doc('products_0').set(...)`을 직접 호출하면 **승인되지 않은 상태에서도 모든 데이터를 수정·삭제할 수 있습니다.** Firestore 규칙은 `isAuth()`만 확인하기 때문입니다.

- 🟡 **[주의] `js/auth.js:101-156` — `checkApproval()`의 "첫 사용자 자동 관리자" 분기**
  `users` 컬렉션이 비어 있을 때(첫 번째 사용자) Auth 계정만 있어도 grade:4(시스템 관리자) 문서를 자동 생성합니다(112-131행). 시스템이 정상 가동 중인 상태에서는 컬렉션이 비어 있을 일이 없으므로 안전하지만, 만약 누군가 실수로 `users` 컬렉션 전체를 비우면 그 직후 가장 먼저 접속한 사람이 자동으로 시스템 관리자가 됩니다. (현실적 위험은 낮으나 인지 필요.)

- 🟡 **[주의] `js/auth.js:360-383` — `initAdminAccount()`의 초기 관리자 이메일/비밀번호 하드코딩**
  `'lemango@gmail.com'`, `'lemango2026!'`가 코드에 평문으로 박혀 있습니다. 누구나 소스를 보면 확인 가능합니다. 다행히 `users` 컬렉션이 비어 있을 때만 동작하지만, 위에 언급한 컬렉션 비움 사고가 났을 때 노출 위험이 있습니다.

- 🟢 **[안전] 로그인 게이트** — `js/main.js:11-12`에서 비로그인 상태에서는 `initApp()` 자체가 호출되지 않아 데이터 로드 패스가 작동하지 않습니다.

- 🟢 **[안전] 세션 지속 정책** — `SESSION` persistence는 탭 닫으면 로그아웃이라 공용 PC 환경에서 적절합니다.

- 🟢 **[안전] 가입 시 원자성 처리** — `js/auth.js:309-330` 가입 도중 Firestore 저장이 실패하면 Auth 계정도 롤백(`cred.user.delete()`)하여 좀비 계정이 생기지 않도록 잘 짜여 있습니다.

- 🟢 **[안전] 비밀번호 강도 검증** — `js/auth.js:301-303` 8자 이상 + 영문+숫자 포함 + 일치 확인까지 클라이언트에서 검증.

## Area 3: 동시 사용 시 데이터 충돌

**종합 등급**: 🔴
**한 줄 요약**: 회사 핵심 데이터(상품·기획·재고·매출)의 거의 모든 쓰기가 "내가 가진 전체 배열을 그대로 덮어쓰는 방식"입니다. 두 명이 동시에 작업하면 나중에 저장한 사람의 변경만 남고 앞 사람 작업은 사라집니다.

### 고위험 쓰기 패턴

#### 1. 상품 데이터 저장 — `js/core.js:89-108` `saveProducts()`
- `State.allProducts` 배열 **전체**를 150개씩 청크로 나눠 `sharedData/products_0`, `products_1`, … 에 통째로 덮어씁니다(batch.set).
- A가 LSWON12345의 가격을 수정하고, B가 동시에 LSWON67890의 재고를 수정하는 시나리오: A의 변경이 먼저 저장되고 B가 1초 뒤 저장하면, B가 보낸 배열에는 A의 가격 변경이 없으므로 **A의 가격 수정이 통째로 사라집니다.** 800개 상품 중 단 1개만 바꿔도 800개 전체를 다시 올리는 구조입니다.
- 영향 받는 호출처:
  - 신규 입고: `js/stock.js:247` `saveSrmStock` 끝
  - 엑셀 일괄 입고: `js/stock.js:353` `confirmStockUpload` 끝
  - 개별 출고: `js/stock.js:474` `submitOutgoing` 끝
  - 카페24 매출 업로드: `js/gonghom.js:663` `confirmGonghomUpload` 끝
  - 사방넷 매출 업로드: `js/sabangnet.js:576` `confirmSabangnetUpload` 끝
  - 엑셀 상품 업로드: `js/excel.js:2593`, `2613`
  - 색상 변경, 휴지통 복원/영구삭제 등 다수

#### 2. 재고 누적 (`p.stock[sz] += qty`) — `js/stock.js:228, 338, 455`
- 메모리상 상품 객체에 직접 `+=`로 누적 후 `saveProducts()`를 호출합니다.
- A가 M사이즈 10개 입고, B가 동시에 M사이즈 5개 입고를 누르면: 두 사람 모두 "기존 재고 0 → +10/+5"를 자기 메모리에서 계산하고, 나중에 저장한 사람의 값(10 또는 5)이 결과로 남습니다. **정답인 15는 어느 누구의 화면에도 남지 않습니다.**
- Firestore 실시간 동기화(`_onProductsChanged`, `js/core.js:432-455`)는 "내가 0.5초 안에 저장한 것이면 무시"하는 로직(`myLastSave + 500`)이 있어, 짧은 시간 안에 양쪽이 거의 동시에 누르면 양쪽 모두 자기 변경만 보고 상대 변경은 받지 않습니다.

#### 3. 매출 누적 (`p.sales[ch] += qty`) — `js/gonghom.js:617, 624` / `js/sabangnet.js:539`
- 카페24와 사방넷 동시 업로드, 또는 두 명이 같은 파일을 동시에 업로드하는 경우 동일한 누적 충돌이 발생합니다.
- 주문번호 중복 검사(`_buildExistingOrderIndex`)는 "이미 저장된" 주문번호만 보므로, 두 사람이 같은 시간에 같은 파일을 업로드하면 **같은 주문이 두 번 합산**될 수 있습니다.

#### 4. 신규기획 — `js/plan.js:9-21` `savePlanItems()` (단일 문서 패턴)
- `State.planItems` 배열 전체를 JSON으로 `sharedData/planItems` 단일 문서에 통째로 저장합니다(`_fsSync('planItems', ...)`).
- A가 새 기획을 추가하고, B가 다른 기획을 추가하는 시나리오: B가 1초 뒤 저장하면 B의 메모리에는 A가 방금 추가한 기획이 없으므로 **A의 기획이 사라집니다.**
- CLAUDE.md에도 "300건 초과 시 Firestore 1MB 한계" 경고가 있으나, 동시 쓰기 손실 위험이 더 즉각적인 문제입니다.

## Area 4: 데이터 예외 처리

**종합 등급**: 🟢
**한 줄 요약**: 대부분의 렌더링 함수가 `?.`(옵셔널 체이닝)와 `|| 0`/`|| '-'` 기본값을 잘 사용합니다. 발견된 위험 지점은 소수이며 발생 빈도도 낮습니다.

### 발견사항

- 🟡 **[주의] `js/dashboard.js:722` — 매출 요약의 "전월" 값이 가짜 데이터**
  ```
  const prev = Math.round(curr * (0.8 + Math.random() * 0.4))
  ```
  매출현황 요약표에서 "전월" 값이 실제 데이터가 아니라 "당월에 0.8~1.2배 곱한 난수"입니다. 새로고침할 때마다 다른 값이 나옵니다. 데모용 placeholder가 그대로 남아있는 것으로 보입니다. 사용자에게 잘못된 비즈니스 판단을 유도할 수 있습니다.

- 🟡 **[주의] `js/dashboard.js:702` — `p._periodSales.toLocaleString()` 의 NaN 가능성**
  `_periodSales`가 위 함수의 reduce 결과로 들어오는데, 만약 모든 `p.sales`가 `undefined`인 신규 상품이 섞이면 `0`이 되어 안전합니다. 다만 만일을 대비해 `(_periodSales || 0)` 가드가 빠져 있어 데이터 마이그레이션 직후에는 `'NaN'` 문자열이 보일 수도 있습니다.

- 🟡 **[주의] `js/register.js:299` — `r.salePrice.toLocaleString()` 조건부 호출**
  `r.salePrice ? r.salePrice.toLocaleString() + '원' : '—'` 로 0/undefined를 모두 `'—'`로 처리합니다. 다만 `r.salePrice`가 문자열로 들어오는 경우(엑셀의 셀 타입 변동) `.toLocaleString`이 없는 객체일 수 있습니다. `Number(r.salePrice).toLocaleString()`이 안전합니다.

- 🟡 **[주의] `js/utils.js:1085-1087` — `parseExcelDate`가 형식이 어긋난 날짜 문자열을 그대로 반환**
  ```
  const s = String(val).trim()
  if (/^\d{4}-\d{2}-\d{2}/.test(s)) return s.slice(0,10)
  return s
  ```
  `"2026/04/01"` 같은 슬래시 형식이 들어오면 그대로 `"2026/04/01"`이 반환되어 이후 `new Date(s)`나 `localeCompare` 비교에서 NaN/이상 정렬 결과를 만들 수 있습니다. 엑셀 셀이 셀형식 오류로 텍스트로 들어오는 경우 발생 가능.

- 🟢 **[안전] `js/utils.js:303-306` — `fmtPrice` / `fmtNum`**
  null/undefined를 `'-'`로 잘 처리합니다. 다만 `NaN`이 들어오면 `'NaN'`이 표시되긴 합니다(현실적으로 거의 발생 안 함).

- 🟢 **[안전] `js/utils.js:312-316` `getThumbUrl`** — `p.images?.sum?.length` 같이 옵셔널 체이닝을 잘 사용하며, 폴백 이미지(`PLACEHOLDER_IMG`)도 모든 호출처에서 적용되어 있어 깨진 이미지 위험 거의 없음.

- 🟢 **[안전] `js/utils.js:305` `getTotalStock` / `getTotalSales`** — `p.stock || {}` 가드로 신규 상품에도 안전.

- 🟢 **[안전] `js/dashboard.js:25-26` 재고/매출 합산** — `p.stock[sz] || 0`, `p.sales?.[pl] || 0` 가드 잘 적용.

- 🟢 **[안전] 카페24 파싱 (`js/gonghom.js:51`) `_toNum(v)`** — `NaN` 체크 명시적으로 들어 있음. 매출 계산 안정.

- 🟢 **[안전] 엑셀 업로드 (`js/excel.js:1350-1356`) `_n(key)`** — `Number(raw) || 0` 패턴으로 NaN/빈 셀 모두 0 처리.
